# References: snowflakes in event tables, Guids in core entities

We carried both `*_id` (internal `Guid`) and `*_discord_id` (Discord snowflake `ulong`) on most rows. Live audit shows the columns serve different roles depending on table family: in core entity tables the `Guid` FK is always populated, but in append-only event tables (`member_events`, `voice_state_events`, `presence_events`, …) the `Guid` is nullable and sometimes `NULL` for rows where user-upsert hadn't completed at insert time — so the snowflake is the only canonical reference. We resolve §P2.3 along that grain: drop `Guid` FKs from event tables (keep snowflakes — they're the natural key from Discord), and drop snowflakes from core entity rows like `ActivityEntity` (keep `Guid` — always resolved). Queries crossing the boundary join `event.user_discord_id = users.discord_id`.
