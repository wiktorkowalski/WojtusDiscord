# Conversational assistant is an agentic LLM loop on Microsoft.Extensions.AI over OpenRouter

**Status**: Accepted (2026-06-19).

The bot gains an interactive side: a member @mentions it (a thread is spawned off the message, per ADR-0003) or DMs it, and it holds a multi-turn conversation that answers questions about the guild from the ingested data and — for admins — performs Discord actions. It is built as a hand-driven **agentic loop** (model → tool calls → model → … → final reply) on **Microsoft.Extensions.AI** (`IChatClient`), with the OpenAI client pointed at OpenRouter (`anthropic/claude-sonnet-4.6` by default, configurable; OpenRouter stays the single provider per ADR-0005). The loop is owned by application code rather than a framework's auto-invocation so that each round is *visible*: the bot posts an interim "looking into that" message, narrates each tool batch, then streams the final answer in place — the Claude-Code cadence the feature exists for. A hard per-turn round cap guards against runaway tool loops.

MEAI was chosen over Semantic Kernel (which both sibling bots — homelab, minecraft — use) and over extending the existing raw-HTTP `OpenRouterClient`. SK's headline feature is auto-invocation, which hides exactly the round seam this feature needs, so SK would run in manual mode anyway — while still requiring a `DelegatingHandler` to inject Claude's `cache_control`/`provider`/`reasoning` body fields, plus the experimental gen_ai OTel switches and a bespoke enrichment processor for Langfuse. Raw HTTP makes those Claude fields first-class but leaves the wire (DTOs, SSE parsing, streamed tool-call assembly) to own forever. MEAI sits one layer below SK (SK 1.x is built on it; `KernelFunction` *is* an `AIFunction`): it deletes the wire plumbing, emits standard gen_ai OpenTelemetry spans Langfuse renders without an enrichment processor, and keeps the Claude-specific body fields (`provider`, `reasoning`, `cache_control`) reachable through `ChatOptions.RawRepresentationFactory` + `JsonPatch` (the bulk `AdditionalProperties` dictionary is silently dropped by the OpenAI adapter, so it is *not* the mechanism — this also pins `OpenAI` >= 2.10.0 and a Langfuse build from Sep-2025 or later, per the verification pass). The one capability SK has out of the box that MEAI lacks — multi-agent orchestration — is not needed for a single assistant and composes on top later if it ever is.

Capabilities and their boundaries are deliberate. **Read/analytics is open to every member**, in allow-listed channels and in DMs; **write-actions are admin-only everywhere** — a flat allow-list gate enforced in code via an un-promptable invocation context (caller identity/permissions are captured out-of-band, never a model parameter), so injection from a member cannot escalate. Opening analytics to everyone makes the guild's already-public history trivially queryable and aggregable, including privately in DMs — a surveillance *convenience* accepted as a reasonable trade for a friend-group guild. Reads are served by a hybrid tool set: curated typed tools for hot paths (meme search reusing the ADR-0005 FTS, leaderboards) plus a guarded `query_database` SQL escape hatch for the long tail. That SQL tool runs under a **restricted, non-superuser SELECT-only Postgres role** (`wojtus_query`; see the implementation note below), distinct from the ingestion writer, with a per-query read-only transaction, statement/lock timeouts, and an app-enforced row/field cap — so an open-to-everyone SQL surface can only ever perform bounded reads of public data, never a write. Irreversible admin actions (role changes, deletes, moderation) are staged behind a Discord confirm button (the sibling bots' deferred-delegate pattern, with confirm restricted to admins) to catch the *model* mis-parsing intent; reversible ones execute immediately.

**Implementation note — `query_database` read-only identity (§4, as built).** The SQL tool does *not* open a separate connection. Because the application's Postgres login is a **superuser** (it needs `CREATE EXTENSION` during migrations), a read-only transaction alone would not stop superuser-only side effects (`pg_read_file`, `pg_terminate_backend`, …) — none of which are data *writes*. So each query runs on EF's own connection inside a transaction that is, before any model SQL, `SET TRANSACTION READ ONLY` **and** `SET LOCAL ROLE wojtus_query` — a `NOSUPERUSER NOLOGIN` role granted `SELECT` on the `public` schema only — then **always rolled back** (resetting both the role and the read-only flag for EF's next use). The built-in `pg_read_all_data` was rejected: it also reads `pg_catalog` (including `pg_authid` password hashes), whereas `wojtus_query` is scoped to `public` and cannot. Defence is layered: the role drop + read-only txn are the boundary; an app-layer single-`SELECT`/`WITH` guard, a client `CommandTimeout` under a server `statement_timeout` + `lock_timeout`, and a row/field-size cap bound the rest; bigint snowflakes are stringified for precision. The role is provisioned by the **`AddConversationQueryRole` EF migration**, guarded to run only where the migrating login can manage roles (`rolsuper OR rolcreaterole`) so a de-privileged login skips it instead of failing startup — until the role exists, `query_database` **fails closed**. The longer-term fix that would retire the per-query role entirely is making the application login itself non-superuser.

**Implementation note — admin actions + tiered confirm (§6, as built).** Action tools are admin-only and the gate is decided entirely out-of-band: `ConversationEventHandler` computes `ConversationContext.IsAdmin` from `ConversationOptions.AdminUserIds` (the event author) and bakes it — plus the reply-surface `ChannelId` — into the per-turn tool closures, so admin-ness is *never* a model parameter and a prompt-injected member (or injected tool-result text) cannot escalate. Each action tool checks `IsAdmin` first, before any Discord call or staging. The Discord writes live behind a stateless singleton `GuildActionService` that **attempts the call and catches** DSharpPlus `UnauthorizedException` (403 / role-hierarchy), `NotFoundException`, and `BadRequestException` into clean strings — no permission preflight. Reversible actions (reaction, pin) run immediately; irreversible ones (grant/remove role, timeout, kick, ban, delete) are staged by a singleton `ConfirmationService` that registers the pending action (an opaque token + a deferred execute delegate capturing only ids + the singleton action service) and posts a confirm/cancel button whose `CustomId` carries the token. A new `ConfirmationComponentHandler` (`IEventHandler<ComponentInteractionCreatedEventArgs>`, the codebase's first interactive component) processes the click: it re-checks the **clicker** (not the requester) against `AdminUserIds` every click — a non-admin click never consumes the token — then `TryClaim`s (TryRemove) the token *before* executing, so a double-click can't run the action twice. Staged actions are in-memory only and dropped on restart; an unclaimed action self-expires after `ConfirmExpirySeconds`. Both singletons reach the `DiscordClient` through a plain **`DiscordClientAccessor`** holder — the client is deliberately **never** registered or forwarded into the DSharpPlus child container, because resolving it there re-enters the client's own construction and deadlocks boot (found the hard way; fixed in PR #251). The `ConfirmationService` store is shared because staging and the click both run in that child container.

Conversation state is a new append-only store keyed by thread/DM-channel Snowflake (per the ADR-0002 grain), recording full turns *including* tool calls and tool results so the agent remembers what it already looked up across a turn (the explicit fix for the amnesia in the minecraft bot, which persists only user/assistant text); replay is a token-aware sliding window.

**Implementation note — conversation store (§5, as specified 2026-07-17 for the harness-deepening epic; validated in `docs/research/conversation-memory-and-retry-policy.md`).** "Recording full turns" must **not** be an opaque serialized-`ChatMessage` blob: `FunctionResultContent.Result` is typed `object?`, so a JSON round-trip rehydrates it as a `JsonElement`, misses the OpenAI adapter's `Result as string` fast-path, and the replayed tool message reaches the provider quote-wrapped — different wire bytes than the model saw live. The store is therefore **normalized columns** at the grain of **one row per `ChatMessage`** (an assistant message bundles its tool calls as a `jsonb` array; each tool result is its own row with the result as plain `text`, rehydrated back into a `string`-typed `Result`). The "token-aware" window likewise cannot be sized by the usage ledger — provider usage is per-request, and Anthropic has no offline tokenizer — so the window budget runs on a locally estimated per-message token count (a GPT-style approximation), with a last-M-messages backstop, never splitting a tool-call/result pair across the boundary. Reasoning content is persisted but stripped on cross-turn replay (thinking-signature staleness). The ledger records per-round *attempts* (retried and mid-stream-failed calls still bill), carries the invoker's id per row so per-user cost sums are a plain `WHERE`, and feeds the soft, alert-only cost caps — enforcement (refusing turns) remains deferred behind the same seam. It is a *separate* store from the ingested `messages`/`message_events` — the bot's own posted replies still flow through the normal ingestion path, so "message" now carries three senses (see CONTEXT.md). Observability is rich Langfuse tracing (each round a generation, each tool a span, tokens + real OpenRouter `usage.cost`) plus a queryable Postgres usage ledger. Rate-limit and budget caps are deliberately *deferred* (matching the sibling bots), but because the open-to-everyone surface makes runaway cost the most likely first follow-up, the architecture leaves a guard seam to add them without rework.

**Implementation note — Langfuse tracing (as verified in prod 2026-07-18, PR #280).** The intended shape — one trace per turn, each round an MEAI gen_ai generation, each tool dispatch a span, all nested under a hand-rolled `conversation.turn` parent — did not hold as first built: `GenerateReplyAsync`/`StreamRoundAsync` are async iterators, and after every `yield return` they resume on the consumer's execution context, where `Activity.Current` is no longer the turn span. Only the round-1 generation nested; every tool span and round-2+ generation exported as its own orphan root trace. The fix is `ReanchorTurnSpan()` — restore `Activity.Current` to the turn activity immediately before anything that starts a child span (each round's chat call, including retry attempts, and the tool-dispatch loop); the same trap applies to any future span started inside an async iterator. The turn span also carries Langfuse's mapped identity attributes: `user.id` = invoker Snowflake (trace userId, per-user usage views), `session.id` = thread/DM-channel Snowflake (turns group per conversation), `langfuse.trace.metadata.*` (invoker display name, guild id, channel id — top-level filterable, unlike the `conversation.*` attribute bag), and `langfuse.trace.input`/`.output` (the member's message / final answer) under the same sensitive-data gate as the MEAI chat client (`EnableSensitiveData || IsDevelopment`). The exporter's resource declares `service.name=discord-event-service` and `langfuse.environment` (lowercased host environment), so dev and prod traces separate inside the one shared Langfuse project both export to. Langfuse prices generations from its own model table, which does not know OpenRouter slugs — a project model definition for `anthropic/claude-sonnet-5` ($2/M in, $10/M out, created via `POST /api/public/models`) makes its native cost columns match the ledger exactly, except that Langfuse cannot see OpenRouter's $0.005/request web-search fee: the `conversation.cost_usd` attribute (real `usage.cost`) remains authoritative, and cost is computed at ingestion, so definitions never reprice existing traces.
